01 · Foundation
Design principles
Seven rules. Every screen must pass every rule. If a screen breaks one, it ships broken.
P1
Money is the loudest thing on screen.
The revenue-at-risk number is always the largest, cleanest type on any view it appears in. Charts, decoration, and chrome bend around it. If another element competes for the eye, it is wrong.
do — set currency in mono, 44–72px, tabular.   don't — dress it with gradients, sparkles, or "delight."
P2
Calm is the default. Urgency is earned.
Color and motion are reserved. A screen with no problems is nearly monochrome. When severity rises, the system brightens — never the reverse. A critical issue is allowed to interrupt; a medium one is not.
do — reserve crimson for critical only.   don't — color-code everything.
P3
One signal color. One job.
Amber signals revenue at risk. Nothing else uses it — not CTAs, not links, not tooltips, not marketing. A user learns this in the first ten seconds and trusts it for the next three years.
do — amber = "money lives here."   don't — amber a "subscribe" button.
P4
Hairlines over cards.
Panels are separated by a 1px hairline or a quiet surface shift — not elevation, not shadow, not rounded boxes floating on a tinted background. Card-chaos is the visual tell of a dashboard that doesn't know what matters.
do — separate with line-default.   don't — nest cards inside cards inside cards.
P5
Numbers live in mono.
Currency, counts, percentages, timestamps, store IDs, transaction references, webhook payloads — all tabular monospace. Prose lives in sans. This creates an instant visual taxonomy: the user's eye finds data without reading.
do — $14,382.40 in mono, tabular.   don't — prose with a number dropped in.
P6
Every screen answers four questions.
Where am I? What has happened? What is the next move? What happens after I take it? If a screen cannot answer one of these in under two seconds of eye-tracking, it is incomplete. This rule governs every state pattern in the system.
do — put the next move within arm's reach.   don't — end a screen without a verb.
P7
The product looks the same when it's working.
A healthy workspace is quiet, almost empty. Resist the urge to fill it with charts, streaks, or celebration. An operator tool that congratulates itself is a tool that has lost the plot. The reward for a clean store is silence.
do — "No issues detected. Last scan 00:14 ago."   don't — confetti, trophies, "great job!"

01.8
Shadow — reserved
Shadows are a last resort. They appear only on floating layers: popovers and modals. Panels and cards do not cast shadows.
Popover — shadow-popover
Modal — shadow-modal
01.9
Motion
One easing curve. Four durations. Nothing overshoots. Nothing bounces. Active scans may pulse at 2s; nothing else animates on idle.
dur-instant80ms · micro-feedback
dur-fast120ms · hovers, state swaps
dur-base180ms · panel transitions
dur-slow280ms · modal open/close
easing · cubic-bezier(0.2, 0, 0, 1) — no rebound
01.10
Focus
Focus is a contract. Every interactive element has a visible focus ring in the signal color — yes, the same amber reserved for money. Focus is the one other thing the system treats with equal seriousness.
outline 2px signal · offset 2px
border + 3px signal @ 14%
Nav link
keyboard only · not on :hover

01.12
In code
Every token is a CSS custom property on :root. Tailwind, styled-components, or vanilla — pick any, but consume the same variables.

06.1
Six regions — the grammar
Every dashboard view in CheckoutLeak composes these regions. Omit one and the screen loses information. Add a seventh and the screen loses focus.
R1 · Primary metric
Revenue at risk — top-left, always.
The largest, cleanest number on the screen. Mono, tabular, signal amber. Subtitle is the delta vs the prior window, colored by direction. This region occupies roughly one quarter of the visible real estate above the fold.
Rule — 48–64px mono · signal color · always labeled with a time window
R2 · Leak queue
Issues, sorted by exposure.
The queue is the product. It sorts by dollar exposure by default, not by severity, not by recency. Every row: severity stripe, severity pill, title with a mono code fragment, sub-line of source + meta, age, and mono currency anchor.
Rule — sort by exposure · severity is color only · currency mono-right
R3 · Next move
One recommendation, upper-right.
The system picks exactly one next action and commits to it — the one that moves the at-risk number down the most. Signal-tinted background, one-line prescription, a primary button that opens the fix path. Never a list of suggestions.
Rule — single prescription · primary button is verb + noun (Open fix path, Replay events)
R4 · Source health
The instrumentation itself.
Each connected source in one row: status dot, name, scope, latency. When a source fails, this region brightens — never flashes, never modals unless critical. The user must trust that if R4 is silent, the signal in R1 is real.
Rule — semantic dot colors · p99 latency mono · critical state stays on screen until acknowledged
R5 · Evidence
Live tail of the raw corpus.
Mono log, timestamps in tertiary, error substrings in severity color. This is where the operator confirms the system is seeing what they're seeing. It is never the main story; it sits below the queue.
Rule — JetBrains Mono 11px · timestamp left · no prose
R6 · Legend / grammar
A small, persistent severity key.
Four rows describing what each severity level means in this product. It lives in a fixed corner — usually lower-right of the sidebar column — and never moves. New operators read it once; everyone else ignores it, which is correct.
Rule — small type · never occupies primary focus · never rewrites itself
06.2
Anti-patterns
Things that look right on a Dribbble shot and fail in production.
Don't
Paint eight metric cards across the top. The user will read zero of them.
Do
One signal metric, three supporting. R1 is the reason they opened the tab.
Don't
Put a donut chart of "issues by category" at the top. Categories don't pay the bills.
Do
Sort the leak queue by exposure. Dollars outrank taxonomies.

07.2
Mobile rules
Mobile is a different product, not a smaller product.
M · R1
Spacing compresses, type does not shrink below 12px.
Padding drops one step: desktop 24 → mobile 16; desktop 16 → mobile 12. Type scale stays. The primary metric stays at 34–38px.
spec — panel padding 16px · section gap 14px · body 13px · meta 10px min
M · R2
Bottom tab bar, never a hamburger.
Four tabs: Leaks, Sources, Recovered, More. Leaks is always the default landing tab. The tab bar is the operator's root navigation; secondary routes live inside tabs.
spec — 4 items · 56px tall + safe-area · active item uses signal color for icon + label
M · R3
Panels stack linearly. No side-by-side on phone.
The desktop 2-column grid collapses to a single column in this order: primary metric → stats row → queue → next move → source health. Source health is always below the fold.
spec — single column · never horizontal scroll · panel boundaries hairline only
M · R4
The primary CTA pins above the tab bar.
The single "resolve top finding" or "open fix path" action floats in the thumb zone. It is never scrolled off-screen. On the all-clear state, the CTA disappears entirely rather than rewording to "scan now."
spec — pinned bottom · 16px inset · 44px min hit · hidden when no action applies
M · R5
Currency truncates with a 'k' or 'm' above $10,000.
On mobile, $14,382.40 renders as $14.3k. Full precision remains available on the detail screen. Truncation is a mobile rule, never a desktop one — never lose digits on desktop.
spec — < $10k → full · $10k–$1m → $14.3k · > $1m → $4.2m
M · R6
Touch targets are 44 × 44px minimum.
This applies to every interactive element — queue rows, toggles, tab items, severity pills. A row smaller than 44px tall gets vertical padding, not a smaller font.
spec — row min-height 56px · tap padding overrides design padding · no 12px-tall controls
M · R7
What collapses, what stays.
Collapses: the workspace side-nav (becomes 'More'), the evidence log (behind a tap), the severity legend (behind Help). Stays persistent: the primary metric, the queue, the source-health dot summary.
spec — persistent: metric, queue, health dots · collapsed: nav, evidence, legend, rules
M · R8
Push notifications match severity.
Critical → interruptive push with amount in the title. High → silent push with badge. Medium / low → no push; digest only. A push notification from CheckoutLeak is an operator promise: "this was worth waking up for."
spec — crit · interrupt + amount · high · silent + badge · med / low · digest
08 · Voice

08Content & UI language

Vault is precise, premium, commercially grounded. It writes like an operator writing for operators. No product jargon, no emotional leakage, no celebration of its own features.

08.1
Voice principles
Six rules. They govern button labels, banners, email, push, marketing, and error strings alike.
V1
Name the money, not the feeling.
"Revenue at risk" not "issues detected." "38 events dropped" not "something went wrong." A number is always more useful than an emotion.
V2
Every sentence carries a next action — or none.
Either tell the user exactly what to do, or say nothing. Don't narrate the system's inner life. "Coverage incomplete. Connect Shopify." is two sentences; both are load-bearing.
V3
Technical words are welcome.
The audience is engineering-literate. "Webhook signature", "3DS step-up", "setup_intent", "dunning" are load-bearing — not jargon. Dumbing them down reads as condescension.
V4
Never apologize. Never celebrate.
No "oops!", no "we're on it", no "great job!", no streaks, no trophies. The system is a serious instrument. It reports and it recommends.
V5
Tense: present or past. Not future-promise.
"38 events dropped." "Scan complete." Not "we'll make sure this never happens" or "your revenue is protected." We report what we see.
V6
Mono words inside sans sentences.
Any technical value — an event name, a domain, a variable — renders in mono inline with the prose. This is a voice rule as much as a typographic one; it signals what the user can click, copy, or act on.
08.2
What we say · what we never say
Two columns. The right column contains real copy we have caught ourselves writing. The left is the rewrite.
Say
Revenue at risk · 30d
Names the commercial unit and the window.
Leak detected · webhook signature
Specific surface, specific failure mode.
Coverage incomplete — Shopify not connected.
States the fact and the missing input.
First scan in progress — 4 min remaining.
Time-bounded, no fluff.
Recommended next move
Not "suggestion." The system picks one.
Fix path · 3 steps
Stepwise, finite, finishable.
Source connected — first scan queued.
Confirms the prior action; declares the next.
Billing requires attention.
Precise. No exclamation mark.
No issues detected · last scan 00:14 ago.
Silence is a valid result.
Never say
"Your insights are ready! 🚀"
Vague noun + celebration + emoji.
"Looks like we found something interesting…"
Coy. Operators don't have time.
"Uh oh! It seems like Stripe is acting up."
Infantilizing; blames the upstream.
"Great job! You've resolved 12 issues this week."
No trophies. The reward is the numbers going down.
"Let's get you set up."
First-person plural; consumer voice.
"We're working on it."
Narrates the system's inner life.
"Don't worry — your revenue is safe with us."
Promises. Never promise.
"Tap here to unlock premium insights"
Marketing leaking into product.
"Welcome back! Here's what's new."
Non-operator voice. No homepage fluff inside the product.

08.4
The canonical vocabulary
Terms the product uses consistently. New features should extend this vocabulary, not invent parallel one.
Revenue at risk · the money that could be lost if an issue isn't fixed within the window.
Leak · a confirmed revenue loss pattern, not a hypothesis.
Finding · one instance of a leak, with ID, severity, exposure.
Source · a connected Stripe account or Shopify store.
Coverage · the portion of revenue surface the system can see.
Scan · a sweep of the corpus that produces findings.
Fix path · the ordered steps to resolve a finding.
Exposure · dollars attributable to a single finding.
Recovered · dollars the user has reclaimed since resolution.
Severity · low · medium · high · critical. Fixed four-level scale.
09
Implementation notes
Closing the system.
I · 01 · Tokens
Ship tokens as CSS custom properties on :root.
Consuming them via Tailwind, styled-components, or raw CSS is equivalent. Do not fork the palette for marketing — marketing consumes the same --signal as product.
I · 02 · Theming
There is no light theme.
A light-mode port would destroy the operator language. If accessibility dictates a higher-contrast variant, build a prefers-contrast: more override on the existing dark palette — increase text primary to oklch(1 0 0), deepen borders to 0.12 alpha.
I · 03 · Review checklist
Five questions to clear before any screen ships.
  1. Is the primary money number the largest thing on the screen?
  2. Is every numeric value set in mono with tabular figures?
  3. Is amber reserved for revenue-at-risk only, nowhere else?
  4. Does the screen answer: where am I, what happened, what's next, what after?
  5. If the workspace is healthy, is the screen quiet?
I · 04 · Scope & evolution
Vault is a working system, not a style guide.
Every token, component, and pattern here compiles to CSS and ships. New features extend the system by adding components that follow these grammars — never by forking the tokens.
